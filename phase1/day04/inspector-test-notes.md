This means your blank-IP validation may not be surfacing the ToolError correctly through FastMCP's notification channel. The client received an error log, not a structured tool error result.

Those are different things. An empty string "" passes schema validation and hits your if not ip check. A missing/unfilled field may be rejected at the schema level before your code runs — because ip is in required: ["ip"] — and FastMCP emits a notification-level error instead of routing it to your ToolError handler.
That's the bug to document: required field missing = schema rejection = notification log, not ToolError. Your validation code never even ran.
